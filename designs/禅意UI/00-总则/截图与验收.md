# 截图与验收 · 操作手册

## 为什么不用预览面板截图

预览面板把页面**交给外部浏览器打开**（面板卡片上写 "Opened in Browser"），面板自身 `document.visibilityState` 恒为 `hidden`、不合成帧，`computer{action:"screenshot"}` 必然超时报 `not compositing frames`。重试、换 tab、重启 dev server 都无效。

另有一类可修的原因：`.claude/launch.json` 若不加 `--strictPort`，vite 在端口被占用时会**静默漂移**（5173→5174），而 preview 代理仍指向自己分配的端口，代理端口上没有页面。现已在 launch.json 里加了 `--strictPort`，并备了 `dev-alt`(5180) 供并发会话使用。

**结论：验收一律用 `scripts/screenshot.mjs`，不要跟预览面板较劲。**

## 用法

```bash
npm run dev            # 或 preview_start 起 dev-alt(5180)
npm run shot -- --shots=home,home:dark --statusbar
```

出图落在 `screenshots/`（已 gitignore），用 Read 工具直接看 PNG。

| 参数 | 作用 |
| --- | --- |
| `--shots=a,b:dark,c@modal` | 一次启动批量出图；`:dark` 指定夜间；`@modal` 表示开抽屉而非切页 |
| `--statusbar` | 叠一条 iPhone 状态栏（59pt），按 `default` 语义把内容整体下推再画，用来判断真机顶部留白 |
| `--nav=holdings` | 先点进某个内页再截 |
| `--modal=quickAdd` | 打开某个抽屉再截（quickAdd / month / liability，或直接给选择器） |
| `--theme=dark` | 单张夜间 |
| `--width= --height=` | 换视口，默认 430×932（iPhone 15 Pro Max）；对定稿图基准可用 390×844 |
| `--url=` | 换地址，默认 http://localhost:5180 |

它用 `puppeteer-core` 驱动**系统已装的 Chrome**，不下载浏览器二进制；跑在独立临时 profile 且 `--disable-extensions`，不碰用户 Chrome 的配置、扩展与登录态。

## 几条踩过的坑

- **批量优先**。每次启停 Chrome 都是一次进程与管道往返，串联多条命令时撞上过外层工具的结果回传故障（`[Tool result missing due to internal error]`，命令其实成功了）。要多张图就用 `--shots` 一次跑完。
- **长命令输出落盘**：`node scripts/screenshot.mjs ... > /tmp/shot.log 2>&1` 再单独读日志，别让管道长时间挂着。
- **每张图独立浏览器上下文**：应用会记住上次停留的页面，共用 profile 时后一张的背景会串成前一张（抽屉截图的背景一度是持仓页而非首页）。脚本已按张隔离。
- **状态栏要下推不要覆盖**：`status-bar-style=default` 时页面从状态栏下方开始；若把状态栏盖在页面上，会把顶部内容压掉、得出「留白正好」的错误结论。
- **DPR=2 下面板截图会裁切**（最多约 800×1733 物理像素）。本脚本不受影响；若不得已用面板截图，可临时 `transform: scale(0.5)` 把整屏压进可捕获区域，注意此时所有 rect 测量都要除以缩放比。

## 验收清单里最容易漏的一项：水平居中

`text-align: center` 只作用于**行内内容**。父级一旦是 flex 容器，子项的排列由 `justify-content`/`align-items` 决定，`text-align` 完全不起作用——首页的 hero 整区就是这样贴在左边的，而 CSS 里明明写着 `text-align: center`。

所以不能靠读 CSS 判断，要实测中心偏移：

```js
const page = document.querySelector('#homePage');
const cs = getComputedStyle(page), pr = page.getBoundingClientRect();
const L = pr.left + parseFloat(cs.paddingLeft);
const R = pr.right - parseFloat(cs.paddingRight);
const C = (L + R) / 2;                      // 内容区中线
const off = (el) => {
  const r = el.getBoundingClientRect();
  return Math.round((r.left + r.right) / 2 - C);   // |偏移| <= 2 视为居中
};
```

同理，纵向也别只信 `margin-top` 写了多少——旧层的 `height`/`min-height`/`flex` 会在特定屏高下改变实际位置。

## 机检：`npm run verify`

```bash
npm run verify                        # 首页
npm run verify -- --page=holdings     # 内页
npm run verify -- --page=quickAdd@modal   # 抽屉
npm run verify -- --page=home --json  # 机读
```

一条命令跑完全部机械性红线，并出日夜两张图。退出码 0 = 全绿。

| 检查项 | 抓什么 |
| --- | --- |
| 水平居中 | 声明了居中意图却没真居中；**父级是列向 flex 且 align-items 非 center 时标记为高危**——hero 整区贴左就是这个特征 |
| 无描边 | 逐边查 border（只有 border-bottom 的元素很容易漏） |
| 无横贯分隔线 | 用 div/伪元素背景做的 rule（宽度≥内容区 90%、高度≤4px）；金线一类不满宽，不误报 |
| 无伪元素装饰 | ::before/::after 带背景色且在绘制（旧刻度线、旧圆点图例） |
| 无按钮底托 | button 有非透明背景（chip 盒） |
| 无紫色残留 | 旧 Swiss iris 色 |
| 数字等宽 | 含数字的叶子节点是否 tabular-nums |
| 单屏 / 横向溢出 | 430×932 下内容高是否超视口、是否横向溢出 |
| 尺寸等比 | 同一元素在 390 与 430 两个视口下的字号比值应≈1.103，写死绝对 px 会被抓出 |
| 掩码不破版 | 点隐私按钮后是否有行溢出或撑高 |

### 两条使用纪律

**机检全绿 ≠ 验收通过。** 它只覆盖机械特征，"像不像定稿图"（构图、字阶、留白节奏、金线位置）必须 Read 图目检。首页那次翻车正是机械指标全对、构图整个错。

**机检自己也会漏、也会误报。** 实际发生过：最初只查 `border-top`，漏掉持仓页页头那条只有 `border-bottom` 的 rule；居中检查最初把顶行图标、月点这类本就并排的元素全报成"未居中"。所以拿一个**已验收通过的页面**（当前是 home）当基准跑一遍，全绿再信它——若基准页都报一堆，那是检查写错了，不是页面错了。
